# Ship a change to an ALPS website

Composable base skill for ALL ALPS sites. Never hardcode repo or folder paths — discover them, and ask the user when ambiguous.

## Quick start (the loop)

1. Locate repo → 2. Branch off `main` → 3. Edit → 4. **Build + verify** → 5. PR → squash-merge → 6. **Poll live domain** until the change appears.

## 1. Locate the repo (discover, don't assume)

- If the user already has a clone open, use it. Otherwise list candidates: `gh repo list AlpsFoundation --limit 100`.
- Clone into a fresh temp dir (e.g. `mktemp -d`), not a fixed path: `git clone https://github.com/AlpsFoundation/<repo>.git`.
- **Ambiguous?** (no repo named, several plausible matches) → AskUserQuestion listing the candidate repos. Don't guess.

## 2. Branch

`git fetch origin -q && git checkout -B <descriptive-branch> origin/main`

## 3. Build & verify — handle the pnpm gotcha

These repos build on Cloudflare with **pnpm**. A clean `pnpm install` can exit 1 with `ERR_PNPM_IGNORED_BUILDS` (esbuild/sharp/workerd), which **fails the Cloudflare deploy silently** — the merge succeeds but the site never updates.

- **The committed fix:** ensure `package.json` has `"packageManager": "pnpm@10.15.1"` (a version that installs without the strict ignored-builds gate). pnpm 11 hard-fails; pnpm 9 errors on a workspace file.
- **NEVER add a `pnpm-workspace.yaml` without a `packages:` field** — it breaks pnpm 9 and is ignored by pnpm 11. If one exists with only `onlyBuiltDependencies`, removing it + the pin is the fix.
- Build with the pinned version and a reusable store (fast, matches CI):
  `npx -y pnpm@10.15.1 install --store-dir /tmp/alps-pnpm-store && npx -y pnpm@10.15.1 build`
- For a quick **local-only** build you may `pnpm approve-builds --all` after install — but that does NOT fix CI; only the `packageManager` pin does.
- **Build must exit 0.** Then grep the built HTML for a string proving your change is present:
  `grep -c "<marker from your change>" dist/client/index.html` (path may be `dist/` — check the adapter).

If the build fails for other reasons, fix and rebuild before shipping. See [REFERENCE.md](REFERENCE.md) for deeper pnpm/Cloudflare notes.

## 4. Ship: PR → merge → verify live

- Commit (end the message with the repo's co-author trailer if configured), `git push -u origin <branch>`.
- `gh pr create --base main --head <branch> --title "…" --body "…"`.
- Merge to repo convention — these repos use **squash** (history reads `Title (#N)`):
  `gh pr merge <n> --squash --delete-branch`
  The post-merge LOCAL checkout may error (e.g. dirty `pnpm-workspace.yaml` regenerated by pnpm); the **remote merge still succeeds** — confirm with `gh pr view <n> --json state,mergeCommit`. Delete the remote branch manually if `--delete-branch` didn't: `git push origin --delete <branch>`.
- **Whether to merge at all** is the user's call unless they've said "ship/deploy/update the site." If unsure → AskUserQuestion (merge now vs leave PR for review).

## 5. Verify the deploy is live (don't trust the merge)

Cloudflare auto-deploys from `main` in ~1–3 min. Confirm with a marker unique to your change:

`bash scripts/poll_live.sh https://<live-domain>/ "<marker>"`

Find the live domain from the repo's `wrangler.jsonc` (`routes[].pattern` / custom_domain) or ask. A false-positive trap: a name may already appear elsewhere (e.g. a schedule entry) — pick a marker that only your change introduces (an asset URL, a new class, new copy).

## Composing
